# AI 冰箱营养助手小程序系统开发文档 V1.0

## 1. 项目概述

### 1.1 项目定位

本系统是一款面向微信小程序平台的 **AI 冰箱营养助手**，围绕食材录入、库存管理、保质期提醒、热量记录、菜谱推荐和一周备菜计划构建完整的饮食管理闭环。

系统核心目标：

```text
AI 快速识别食材
        ↓
确认入库
        ↓
库存批次管理
        ↓
临期提醒
        ↓
热量记录 / 菜谱推荐 / 一周备菜
```

项目以生产环境为目标进行设计，要求具备：

```text
模块清晰
数据结构稳定
接口边界明确
AI 输出可控
用户数据隔离
后续可扩展
部署和维护成本可控
```

## 2. 技术栈设计

### 2.1 前端技术栈

本项目不使用 uni-app，采用原生微信小程序开发。

| 层级       | 技术                                   |
| ---------- | -------------------------------------- |
| 小程序框架 | 原生微信小程序                         |
| 开发语言   | TypeScript                             |
| 页面结构   | WXML + WXSS + TS                       |
| UI 组件    | 自定义组件为主                         |
| 辅助组件库 | TDesign Miniprogram / WeUI Miniprogram |
| 状态管理   | 页面状态 + 全局 Store 封装             |
| 构建工具   | 微信开发者工具                         |
| 样式规范   | 全局 design tokens + 组件级 WXSS       |

TDesign Miniprogram 是腾讯开源的微信小程序组件库，适合用于按钮、弹窗、日期选择器、Toast、Picker 等基础交互组件；WeUI Miniprogram 更贴近微信原生视觉体验，也可以作为轻量组件补充。生产项目中不建议全量依赖 UI 组件库，应以自定义业务组件为主，避免视觉风格臃肿。([GitHub](https://github.com/Tencent/tdesign-miniprogram?utm_source=chatgpt.com))

------

### 2.2 后端技术栈

| 层级     | 技术                                       |
| -------- | ------------------------------------------ |
| 后端架构 | 微信云开发 CloudBase                       |
| 服务逻辑 | 云函数                                     |
| 数据库   | 云开发数据库                               |
| 文件存储 | 云存储                                     |
| AI 调用  | 云函数统一封装                             |
| OCR 能力 | 云函数调用 OCR 服务                        |
| 权限控制 | openid + 云数据库权限规则 + 云存储安全规则 |
| 日志监控 | 云函数日志 + 业务日志集合                  |

微信云开发适合小程序生产环境快速落地，云函数可作为服务端业务逻辑入口；小程序调用云函数时会携带用户 openid，云函数中也可以通过 `cloud.getWXContext()` 获取调用上下文，用于用户身份隔离。([腾讯云开发 CloudBase](https://docs.cloudbase.net/cloud-function/how-use?utm_source=chatgpt.com))

云存储和数据库必须配置安全规则，发布前需要避免公开读写规则导致数据泄露。CloudBase 官方文档明确说明，安全规则用于控制客户端发起的读写权限，生产发布前应评估规则安全性。([腾讯云开发 CloudBase](https://docs.cloudbase.net/storage/security-rules?utm_source=chatgpt.com))

### 2.3 AI 能力设计

AI 不直接写入数据库，只负责识别、解析、推荐和生成草稿。

AI 能力分为四类：

```text
1. 食材入库解析 AI
2. 饮食热量解析 AI
3. 菜谱结构化解析 AI
4. 一周备菜计划生成 AI
```

AI 输出必须固定 JSON Schema，由云函数进行校验、清洗和补全。所有 AI Key、OCR Key、模型配置均放在云函数环境变量或云端配置中，禁止暴露在小程序前端。

CloudBase 文档中也提示，小程序场景可使用基础库内置的云开发 AI 能力，避免额外增加包体积和配置服务器域名；本项目可根据实际模型接入成本选择 CloudBase AI、混元、DeepSeek、通义或其他模型接口，但统一通过云函数适配层封装。([腾讯云开发 CloudBase](https://docs.cloudbase.net/ai/sdk-reference/init?utm_source=chatgpt.com))

## 5. 功能模块设计

## 5.1 AI 入库模块

### 5.1.1 入库方式

系统支持三种入库方式：

```text
1. 拍照 / 上传小票
2. 粘贴购物清单
3. 手动自然语言输入
```

示例输入：

```text
今天买了鸡蛋一板，牛奶两盒，鸡胸肉500克，番茄4个。
```

AI 输出结构：

```json
{
  "items": [
    {
      "food_name": "鸡蛋",
      "original_name": "鸡蛋一板",
      "category": "蛋奶类",
      "quantity": 1,
      "unit": "板",
      "storage_location": "cold",
      "purchase_date": "2026-05-20",
      "expire_days": 30,
      "expire_date": "2026-06-19",
      "expire_source": "ai_inferred",
      "confidence": 0.88,
      "note": "未识别到明确保质期，按默认规则推测"
    }
  ],
  "warnings": []
}
```

### 5.1.2 入库确认机制

AI 识别结果不得直接写入库存，必须进入确认页。

确认页允许用户修改：

```text
食材名称
数量
单位
分类
存放位置
购买日期
到期日期
备注
```

用户点击“确认入库”后，系统才写入库存批次表。

## 5.2 库存管理模块

### 5.2.1 批次管理原则

库存必须按批次管理，不能只按食材名称聚合。

错误设计：

```text
鸡蛋：20个
```

正确设计：

```text
鸡蛋
├── 批次1：10个，2026-05-20 入库，2026-06-19 到期
├── 批次2：10个，2026-05-26 入库，2026-06-25 到期
```

前端展示时可以合并显示：

```text
鸡蛋
总库存：20个
最近到期：2026-06-19
```

点进详情页后展示批次。

------

### 5.2.2 库存状态

```text
normal：正常
consumed：已消耗
discarded：已丢弃
```

过期状态建议由 `expire_date` 动态计算，不建议定时强制修改库存状态。

------

### 5.2.3 临期规则

默认规则：

```text
距离到期日 > 3 天：正常
距离到期日 <= 3 天：临期
到期日当天：今日到期
超过到期日：已过期
```

后续可支持用户自定义提醒天数。

## 5.3 卡路里模块

### 5.3.1 页面设计目标

卡路里页面参考轻量健康类 App 的设计方式，核心展示：

```text

```

页面结构：

```text

两侧：已摄入 / 运动消耗

```

------

### 5.3.2 热量计算原则

AI 负责识别用户吃了什么，系统负责计算热量。

```text
用户输入
  ↓
AI 解析食物和克数
  ↓
匹配食物营养库
  ↓
系统计算热量和营养素
  ↓
用户确认记录
```

不建议让 AI 直接给最终热量结果，因为 AI 生成结果存在不稳定性。生产环境应以结构化营养数据库作为计算依据。

------

### 5.3.3 用户营养目标计算

用户需要填写：

```text
性别
出生日期
身高
当前体重
目标体重
目标类型：减脂 / 维持 / 增肌
活动水平：久坐 / 轻度 / 中度 / 高活动
```

基础代谢可采用 Mifflin-St Jeor 公式作为估算方式。该公式在原始研究中给出了男女不同的静息能量消耗计算表达式，也常用于健康成年人能量需求估算。([PubMed](https://pubmed.ncbi.nlm.nih.gov/2305711/?utm_source=chatgpt.com))

男性：

```text
BMR = 10 × 体重kg + 6.25 × 身高cm - 5 × 年龄 + 5
```

女性：

```text
BMR = 10 × 体重kg + 6.25 × 身高cm - 5 × 年龄 - 161
```

总消耗：

```text
TDEE = BMR × 活动系数
```

活动系数：

| 活动水平 | 系数  |
| -------- | ----- |
| 久坐     | 1.2   |
| 轻度活动 | 1.375 |
| 中度活动 | 1.55  |
| 高活动   | 1.725 |

目标热量：

| 目标     | 计算方式         |
| -------- | ---------------- |
| 维持     | TDEE             |
| 轻度减脂 | TDEE - 300       |
| 标准减脂 | TDEE - 500       |
| 增肌     | TDEE + 200 ~ 300 |

系统应避免引导极端低热量饮食。NIH 的体重控制资料提到，减重可通过每日摄入低于消耗约 500 kcal 来实现约每周 1 磅的减重；CDC 也强调减重与热量摄入和身体活动相关，应建立合理、可持续的目标。([NIH News in Health](https://newsinhealth.nih.gov/2022/12/healthy-weight-control?utm_source=chatgpt.com))

## 5.4 菜谱模块

### 5.4.1 菜谱上传

用户可上传文字菜谱、图片菜谱或手动创建菜谱。

AI 解析后生成菜谱草稿：

```json
{
  "recipe_name": "番茄炒蛋",
  "tags": ["家常菜", "快手菜"],
  "cooking_time": 10,
  "difficulty": "easy",
  "ingredients": [
    {
      "name": "番茄",
      "quantity": 2,
      "unit": "个",
      "required": true
    },
    {
      "name": "鸡蛋",
      "quantity": 3,
      "unit": "个",
      "required": true
    }
  ],
  "steps": [
    "鸡蛋打散",
    "番茄切块",
    "先炒鸡蛋",
    "再炒番茄",
    "混合调味"
  ]
}
```

用户确认后保存到菜谱库。

------

### 5.4.2 库存匹配菜谱

推荐排序：

```text
1. 能消耗临期食材
2. 库存匹配度高
3. 符合当前剩余热量
4. 制作时间短
5. 用户收藏或常做
```

菜谱推荐展示：

```text
番茄炒蛋
已有：番茄、鸡蛋
缺少：无
预计热量：320 kcal
推荐理由：食材齐全，制作时间短
```

## 10. 权限与安全设计

### 10.1 身份隔离

所有业务数据必须携带：

```text
user_id = openid
```

所有数据库查询必须限制：

```js
where({
  user_id: openid
})
```

禁止前端传入任意 user_id 后直接作为查询条件。

------

### 10.2 AI Key 安全

禁止在小程序端出现：

```text
API_KEY
SECRET_KEY
MODEL_KEY
OCR_KEY
```

所有第三方服务调用必须在云函数中完成。

------

### 10.3 图片安全

小票图片、菜谱图片、饮食图片上传至云存储后，应配置访问权限。

建议策略：

```text
用户只能访问自己的文件
AI 解析完成后可设置定期清理
敏感图片不长期公开存储
```

------

### 10.4 数据校验

所有写入数据库的数据必须经过云函数校验。

校验内容：

```text
字段是否存在
字段类型是否正确
数量是否为正数
日期是否合法
枚举值是否合法
用户是否有权限操作该数据
```

# 18. 项目开发原则

本系统开发需要遵守以下原则：

```text

2. 库存必须按批次管理
3. 热量计算必须基于营养数据库，不直接采用 AI 估算结果
4. 用户数据必须按 openid 隔离
5. 前端不保存任何敏感 Key
6. 页面只展示核心信息，复杂操作进入详情页
7. 先保证 V1 数据闭环，再扩展菜谱和备菜
8. 所有模块通过云函数统一访问数据库
```

# 19. 最终架构结论

本项目采用：

```text
原生微信小程序 + TypeScript
微信云开发 + 云函数 + 云数据库 + 云存储
云函数统一封装 AI / OCR / 业务逻辑
库存、卡路里、菜谱、计划模块分层设计
```

```text
AI 入库
  ↓
库存管理
  ↓
临期提醒
  ↓
饮食记录
  ↓
热量分析
  ↓
菜谱推荐
  ↓
一周备菜
```

核心技术原则：

```text
轻前端
稳后端
强校验
可扩展
AI 结构化
数据可追踪
生产可维护
```
